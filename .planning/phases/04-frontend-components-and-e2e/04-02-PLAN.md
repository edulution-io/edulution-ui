1. **global-setup.ts** — Authenticate 4 roles via Keycloak, save storageState:
   - Import `chromium` from `@playwright/test`
   - Import `dotenv` and load `.env.e2e`
   - Create `apps/e2e/auth/` directory if it doesn't exist (use `mkdirSync`)
   - For each role (admin, teacher, student, parent):
     a. Launch browser
     b. Navigate to `${E2E_BASE_URL}/login`
     c. Fill username input (data-testid: `test-id-login-page-username-input`) with `E2E_{ROLE}_USER`
     d. Fill password input (data-testid: `test-id-login-page-password-input`) with `E2E_{ROLE}_PASS`
     e. Click submit button (data-testid: `test-id-login-page-submit-button`)
     f. Wait for navigation to dashboard URL pattern (`**/dashboard/**`)
     g. Save storageState to `apps/e2e/auth/{role}.storageState.json`
     h. Close browser
   - Handle errors gracefully: if env var is missing, log warning and skip that role (don't fail entire setup)
   - Add a timeout per login attempt (30 seconds) to prevent hanging
   - Export as default function with `FullConfig` parameter

2. **fixtures/auth.fixture.ts** — Custom Playwright fixture extending base test:
   - Export a `test` object that extends `@playwright/test`'s base test
   - Add custom fixtures for each role:

     ```typescript
     import { test as base, type Page } from '@playwright/test';
     import path from 'path';

     type AuthFixtures = {
       adminPage: Page;
       teacherPage: Page;
       studentPage: Page;
       parentPage: Page;
     };

     export const test = base.extend<AuthFixtures>({
       adminPage: async ({ browser }, use) => {
         const context = await browser.newContext({
           storageState: path.resolve(__dirname, '../auth/admin.storageState.json'),
         });
         const page = await context.newPage();
         await use(page);
         await context.close();
       },
       teacherPage: async ({ browser }, use) => {
         const context = await browser.newContext({
           storageState: path.resolve(__dirname, '../auth/teacher.storageState.json'),
         });
         const page = await context.newPage();
         await use(page);
         await context.close();
       },
       studentPage: async ({ browser }, use) => {
         const context = await browser.newContext({
           storageState: path.resolve(__dirname, '../auth/student.storageState.json'),
         });
         const page = await context.newPage();
         await use(page);
         await context.close();
       },
       parentPage: async ({ browser }, use) => {
         const context = await browser.newContext({
           storageState: path.resolve(__dirname, '../auth/parent.storageState.json'),
         });
         const page = await context.newPage();
         await use(page);
         await context.close();
       },
     });

     export { expect } from '@playwright/test';
     ```

   - This allows tests to use `test('my test', async ({ adminPage }) => { ... })` for pre-authenticated contexts

Create 9 Page Object Model classes. Each class encapsulates page-specific locators and actions.

**BasePage.ts** — Abstract base class:

- Constructor takes `Page` from Playwright
- `navigateTo(path: string)` — navigate to baseURL + path
- `waitForPageLoad()` — wait for network idle or DOM content loaded
- `getByTestId(testId: string)` — shorthand for `this.page.getByTestId(testId)`
- `generateUniqueId()` — returns `${Date.now()}-${Math.random().toString(36).slice(2, 8)}` (per locked decision: unique IDs for all created entities)

**LoginPage.ts** extends BasePage:

- `goto()` — navigate to `/login`
- `usernameInput()` — locator for `test-id-login-page-username-input`
- `passwordInput()` — locator for `test-id-login-page-password-input`
- `submitButton()` — locator for `test-id-login-page-submit-button`
- `loginAs(username: string, password: string)` — fill both fields + click submit + wait for navigation
- `qrCodeButton()` — locator for QR code toggle button
- `expectOnLoginPage()` — assert URL contains `/login`

**DashboardPage.ts** extends BasePage:

- `goto()` — navigate to `/dashboard`
- `expectOnDashboard()` — assert URL contains `/dashboard`
- `getSidebarNav()` — return SidebarNav POM instance

**SidebarNav.ts** extends BasePage:

- `navigateToSurveys()` — click surveys sidebar item, wait for navigation
- `navigateToFileSharing()` — click file sharing sidebar item
- `navigateToMail()` — click mail sidebar item
- `navigateToConferences()` — click conferences sidebar item
- `navigateToSettings()` — click settings sidebar item
- `logout()` — find and click logout action

**SurveyEditorPage.ts** extends BasePage:

- `goto()` — navigate to survey editor
- `createSurvey(name: string)` — interact with survey creation flow (will need data-testids)
- `publishSurvey()` — save/publish actions
- `deleteSurvey(name: string)` — select and delete a survey from list

**FileSharingPage.ts** extends BasePage:

- `goto(webdavShare?: string)` — navigate to file sharing page
- `uploadFile(filePath: string)` — trigger file upload
- `breadcrumb()` — locator for breadcrumb
- `fileTable()` — locator for file table
- `selectFile(fileName: string)` — click to select a file
- `deleteSelectedFiles()` — delete action for selected files

**ConferencePage.ts** extends BasePage:

- `goto()` — navigate to conferences
- `createConference(name: string)` — open create dialog, fill form, submit
- `joinConference(name: string)` — click join on existing conference
- `deleteConference(name: string)` — select and delete

**SettingsPage.ts** extends BasePage:

- `goto()` — navigate to user settings
- `changeTheme(theme: string)` — change theme setting
- `changeNotificationPreference()` — toggle notification preference

**AppStorePage.ts** extends BasePage:

- `goto()` — navigate to app store / settings page
- `browseApps()` — list visible apps
- `installApp(appName: string)` — install an app
- `configureApp(appName: string)` — open app configuration

NOTE: Some POM methods will use generic locator strategies (getByRole, getByText) since not all components have data-testid attributes yet. E2E tests in Plan 04-03 may need to add data-testid attributes to components as needed — this is expected and the POM methods should document which selectors they expect.
