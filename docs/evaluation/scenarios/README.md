# Test Scenarios for Event Logging System

This document describes the coherent test scenarios generated by the demo data loader. These scenarios simulate realistic user workflows for evaluating the recommendation engine.

## Overview

Scenarios are predefined sequences of events that represent common user activity patterns. Each scenario:

1. Has a unique identifier (e.g., `SC-EMAIL-AWAIT-abc12345`)
2. Includes multiple related events with the same `correlation_id`
3. Maps to an expected recommendation class
4. Uses provenance metadata (`_data_source: "demo"`, `_scenario_id`) for traceability

## Provenance Metadata

All demo events include metadata for identification:

```json
{
  "metadata": {
    "_data_source": "demo",
    "_scenario_id": "SC-EMAIL-AWAIT-abc12345"
  }
}
```

| Field | Format | Description |
|-------|--------|-------------|
| `_data_source` | `"demo"` | Identifies event as demo-generated |
| `_scenario_id` | `SC-{TYPE}-{random8}` | Links events to their scenario |

## Scenario Types

### 1. Email Thread Awaiting Reply (`email-thread-awaiting-reply`)

**ID Pattern:** `SC-EMAIL-AWAIT-{random8}`

**Expected Recommendation:** `communication`

**Narrative:** User receives an email, replies, then receives a follow-up that needs a response.

**Event Sequence:**

| Order | Type | Source | Description |
|-------|------|--------|-------------|
| 1 | `mail.thread_created` | mail | Thread initiated by colleague |
| 2 | `mail.received` | mail | Initial email received (72h ago) |
| 3 | `mail.sent` | mail | User's reply (48h ago) |
| 4 | `mail.received` | mail | Follow-up awaiting response (24h ago) |

**Validation:**

```bash
# Check for awaiting-reply recommendation
curl -s http://localhost:3001/recommendations/{userId} | jq '.[] | select(.class == "communication")'

# Verify thread appears in awaiting communications
curl -s http://localhost:3001/events/communications/{userId} | jq '.awaiting'
```

### 2. Meeting Preparation (`meeting-preparation`)

**ID Pattern:** `SC-MTG-PREP-{random8}`

**Expected Recommendation:** `meeting`

**Narrative:** User has an upcoming meeting, receives agenda via email, creates preparation notes.

**Event Sequence:**

| Order | Type | Source | Description |
|-------|------|--------|-------------|
| 1 | `calendar.event_created` | caldav | Meeting scheduled (1.5h from now) |
| 2 | `mail.received` | mail | Agenda email with attachment (2h ago) |
| 3 | `file.created` | files | User creates meeting-prep-notes.md (1h ago) |

**Validation:**

```bash
# Check for meeting-prep recommendation
curl -s http://localhost:3001/recommendations/{userId} | jq '.[] | select(.class == "meeting")'

# Verify meeting appears in upcoming calendar
curl -s http://localhost:3001/events/calendar/{userId} | jq '.meetings'
```

### 3. Chat Discussion with Action (`chat-discussion-action`)

**ID Pattern:** `SC-CHAT-DISC-{random8}`

**Expected Recommendation:** `focus`

**Narrative:** Colleague asks a question in chat, user responds and shares a relevant file.

**Event Sequence:**

| Order | Type | Source | Description |
|-------|------|--------|-------------|
| 1 | `chat.message_received` | chat | Question with mention (2h ago) |
| 2 | `chat.message_sent` | chat | User's response (1.5h ago) |
| 3 | `file.shared` | files | User shares relevant-doc.pdf (1.25h ago) |

**Validation:**

```bash
# Check for focus recommendation
curl -s http://localhost:3001/recommendations/{userId} | jq '.[] | select(.class == "focus")'

# Verify file events are recorded
curl -s http://localhost:3001/events/state/{userId} | jq '.counts_24h | to_entries | .[] | select(.key | startswith("file."))'
```

## Running Scenarios

### Generate Demo Data

```bash
# Full demo with all scenarios
npx tsx apps/api/src/events/cli/load-demo-data.ts --verbose

# Minimal demo (scenarios only)
npx tsx apps/api/src/events/cli/load-demo-data.ts --threads 0 --channels 0 --files 0 --calendar-events 0

# Dry run to preview events
npx tsx apps/api/src/events/cli/load-demo-data.ts --dry-run --verbose
```

### Verify Scenario Events

```bash
# List all scenario IDs from generated events
npx tsx apps/api/src/events/cli/load-demo-data.ts --dry-run 2>&1 | grep "SC-"
```

### Clear and Reload

```bash
# Clear existing demo data (Redis CLI)
redis-cli KEYS "events:*" | xargs -r redis-cli DEL
redis-cli KEYS "state:*" | xargs -r redis-cli DEL
redis-cli KEYS "reco:*" | xargs -r redis-cli DEL

# Reload demo data
npx tsx apps/api/src/events/cli/load-demo-data.ts
```

## Event Structure Alignment

Demo events match production event structure exactly:

| Field | Demo | Production |
|-------|------|------------|
| `event_id` | Full UUID v4 | Full UUID v4 |
| `object.object_ref` | Real path format | Real path format |
| `correlation_id` | `{source}-corr-{uuid}` | `{source}-corr-{uuid}` |
| `metadata` | Includes `_data_source` | No provenance fields |

## Adding New Scenarios

To add a new scenario:

1. Create a new method in `DemoDataGenerator`:

```typescript

  const scenarioId = `SC-MYTYPE-${randomUUID().slice(0, 8)}`;
  const events: Event[] = [];
  const correlationId = `source-corr-${randomUUID()}`;

  // Build events with addProvenanceMetadata(builder, scenarioId)

  return {
    scenarioId,
    scenarioType: 'my-new-scenario',
    events,
    expectedRecommendation: 'planning',
  };
}
```

2. Add to `generateScenarios()`:

```typescript
scenarios.push(this.generateScenarioMyNewScenario(primaryUser));
```

3. Add corresponding rule in `rule-engine/rules/` if needed

4. Document the scenario in this file

## Troubleshooting

### No Recommendations Generated

1. Ensure aggregation worker is running
2. Check that events were published (not deduplicated)
3. Verify time-based conditions (e.g., meeting within 2 hours)

### Scenario Events Missing

1. Check `--no-scenarios` flag is not set
2. Verify Redis connection
3. Run with `--verbose` for detailed output

### Incorrect Recommendation Class

1. Verify scenario event timestamps are correct
2. Check rule thresholds in RuleEngine
3. Ensure event metadata is complete
